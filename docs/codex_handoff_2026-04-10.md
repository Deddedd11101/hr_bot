## Обновление 2026-04-11

Этот блок фиксирует изменения после отладки Telegram-привязки и автозапуска сценариев.

### Backend

- Исправлена ошибка `NameError: set_public_chat_handle is not defined`, из-за которой сохранение карточки могло падать с `500`.
- `POST /api/employees/{id}` теперь не затирает технический Telegram `chat_id`, если frontend не прислал это поле.
- `/start` в Telegram теперь сначала ищет существующую карточку по публичному `@username`, затем привязывает к ней реальный числовой `chat_id`.
- Если карточка по `@username` не найдена, бот создаёт новую карточку как кандидата:
  - `employee_stage = candidate`;
  - `candidate_work_stage = testing`.
- Добавлена защита от дублирования числового Telegram `chat_id` между карточками.
- Логика запуска сценариев теперь учитывает новый `employee_scope`:
  - `all` — для всех сотрудников и кандидатов;
  - `employees` — только для сотрудников не в статусе `candidate`;
  - `candidates` — только для кандидатов.
- Встроенный сценарий `Подбор и найм` (`recruitment_hiring`) переведён на `employee_scope = candidates`.
- Встроенные адаптационные сценарии (`first_day`, `first_week`, `mid_probation`, `end_probation`, `mid_feedback`, `end_feedback`, `end_summary`) переведены на `employee_scope = employees`.
- SQLite schema-guard в `app/database.py` добавляет колонку `scenario_templates.employee_scope`, если её нет, и выставляет базовые значения для встроенных сценариев.
- `tools/scenario_portability.py` теперь переносит `employee_scope`, чтобы сценарии, созданные на стенде, не теряли аудиторию при export/import.

### Текущая логика сценариев

- `Выбор кнопками` убран из нового UX как отдельный полезный тип.
- Основные опоры:
  - обычный шаг;
  - `Ветвление`;
  - `Цепочка шагов`.
- Ветка может содержать ещё одно ветвление.
- Внутрь такой ветки можно проваливаться.
- Цепочка внутри ветки работает и шаги цепочки можно добавлять.
- Автозапуск после регистрации в боте теперь должен фильтровать сценарии по аудитории:
  - кандидат получает сценарии с `employee_scope = candidates` или `all`;
  - сотрудник получает сценарии с `employee_scope = employees` или `all`;
  - приоритет определения — уже найденная карточка по Telegram username/chat id, а не сам username.

### Что ещё не добито по сценариям

- теги для ответа пользователя:
  - `{response_text}`;
  - `{response_file_name}`;
- более структурный UX для редактирования веток;
- reorder глубже основного потока:
  - ветки;

### Что изменили в карточке сотрудника

Нужно помнить главное:

- `ID пользователя в канале` и `Публичный Telegram @username` теперь разделены;
- `@username` хранится отдельно и не должен ломать запуск сценария;
- запуск сценария опирается на числовой `chat_id` из диалога с ботом;
- если числового `chat_id` нет, запуск должен возвращать понятную ошибку, а не падать.
- числовой `chat_id` больше не редактируется вручную в React-карточке;
- `chat_id` должен приходить из Telegram после `/start`;
- публичный `@username` можно внести вручную, чтобы бот при `/start` нашёл заранее созданную карточку и привязал к ней `chat_id`.
