# UI Design Guidelines

Этот документ фиксирует текущие визуальные и UX-принципы HR Bot, чтобы новые экраны и доработки не расползались по стилям.

Он описывает не абстрактный дизайн-манифест, а то, что уже является фактическим контрактом проекта.

## Текущий UI-стек

Новый frontend-слой строится на:

- React 18
- Vite
- Tailwind CSS v4
- Radix UI primitives
- shadcn-style локальных `ui`-компонентах из `frontend/src/components/ui`

Ключевые зависимости:

- `@radix-ui/react-collapsible`
- `@radix-ui/react-popover`
- `@radix-ui/react-scroll-area`
- `@radix-ui/react-separator`
- `class-variance-authority`
- `tailwind-merge`
- `lucide-react`

Практическое правило:

- новые React-экраны и интерактивные UI-блоки строим через Tailwind + локальные `ui`-примитивы;
- не тянем второй визуальный фреймворк;
- не смешиваем случайные сторонние компонентные библиотеки поверх текущего стека.

## Визуальное направление

Проект сейчас держится на спокойной светлой HR-палитре, а не на агрессивной SaaS-эстетике.

Базовые токены из `frontend/src/index.css`:

- `--color-background: #f4f7f4`
- `--color-panel: #ffffff`
- `--color-panel-muted: #f8fbf8`
- `--color-border: rgba(23, 44, 38, 0.1)`
- `--color-foreground: #17312a`
- `--color-muted-foreground: #587066`
- `--color-accent: #1f7a61`
- `--color-accent-foreground: #f7fffb`
- `--color-danger: #b33b4e`

Смысл этого направления:

- фон мягкий, почти бумажный;
- панели светлые и воздушные;
- основной акцент зелёный;
- ошибки красные, но без кислотности;
- контраст читаемый, без тяжёлой чёрно-белой жёсткости.

## Общие принципы интерфейса

### 1. Спокойная, широкая компоновка

Мы не делаем интерфейс “плотным админским Excel”.

Нужно:

- оставлять воздух между секциями;
- использовать широкие панели и спокойные отступы;
- не набивать экран мелкими контролами без иерархии.

### 2. Панели и поверхности

Основные рабочие блоки должны выглядеть как мягкие панели, а не как голые прямоугольники.

Нужно:

- светлый `panel` фон;
- мягкая рамка;
- скругления среднего или крупного радиуса;
- лёгкая тень, а не тяжёлый card-shadow.

### 3. Hover и focus должны быть заметны, но не агрессивны

У проекта уже есть характерный паттерн:

- на hover элементы чуть меняют фон;
- у интерактивных контролов скругление часто увеличивается;
- `focus-visible` подсвечивается через мягкий accent ring.

Это уже видно в локальных `Button` и `Input`.

Нужно:

- использовать плавные `transition-all duration-200`;
- применять accent ring для keyboard focus;
- не делать резких прыжков, сильных масштабирований и “дерганых” hover-эффектов.

## Паттерны компонентов

### Buttons

Источник: `frontend/src/components/ui/button.tsx`

Текущий контракт:

- форма компактная и округлая;
- на hover допускается увеличение радиуса;
- primary-кнопка на accent-фоне;
- secondary/outline-кнопки светлые с границей;
- `ghost` используется для второстепенных действий.

Нужно:

- использовать локальный `Button`, а не писать новую кнопку с нуля;
- для primary-действий держать accent;
- destructive-действия вести через `danger`.

Не нужно:

- заводить произвольные новые button-цвета без причины;
- смешивать несколько “главных” CTA в одном блоке.

### Inputs / textareas / selects

Источник: `frontend/src/components/ui/input.tsx`

Текущий контракт:

- белый фон;
- мягкая граница;
- скругление около `10px`;
- на hover форма чуть “смягчается” сильнее;
- на focus используется accent ring.

Нужно:

- держать поля достаточно крупными для ручной админской работы;
- избегать слишком низких инпутов;
- не делать визуально шумных градиентов и толстых borders.

### Scroll areas

Для сложных рабочих поверхностей используется Radix Scroll Area.

Нужно:

- в больших workspace и split-layout использовать независимые scroll-зоны;
- не полагаться на один общий скролл на всю страницу там, где есть боковые панели и рабочее полотно.

### Popover / dropdown / floating panels

Popover-контролы уже являются частью UX нового workspace.

Нужно:

- использовать Radix-based popover/select-поведение;
- держать popover визуально в том же тоне, что и остальные панели;
- не вводить чужой паттерн dropdown с другим поведением клавиатуры и фокуса.

## Sidebar и shell

Источник: `app/templates/base.html`

Текущий shell-паттерн:

- навигация находится слева;
- sidebar свернут по умолчанию;
- sidebar раскрывается по `hover` и `focus`;
- контент справа плавно подстраивается под состояние shell.

Это уже является source of truth.

Нужно:

- сохранять левый collapsible sidebar как основной shell;
- не возвращаться к верхней навигационной шапке без отдельного архитектурного решения;
- при добавлении новых разделов использовать тот же nav-item паттерн.

Для nav-item важно:

- иконка + label;
- активное состояние читается сразу;
- в collapsed-state иконка остаётся понятной сама по себе;
- раскрытие должно работать и мышью, и клавиатурой.

## Поведение веток, списков и рабочих экранов

Для новых рабочих экранов проекта характерны:

- list/detail композиция;
- независимые scroll-зоны;
- широкая рабочая область;
- мягкая визуальная иерархия без перегруза cards внутри cards.

Нужно:

- в сложных экранах думать про ритм: список, контент, детали;
- не превращать каждую сущность в отдельную громоздкую карточку;
- избегать визуальной дробности и чрезмерного числа маленьких рамок.

### Workspace v2 сценариев

Для `/app/flows/workspace-v2` текущая композиция считается базовой:

- слева список сценариев;
- по центру breadcrumbs и список шагов/веток;
- справа панель деталей выбранного шага/ветки/chain-step.

Настройки сценария не должны разрывать центральный поток шагов. Для них используется отдельная кнопка `Настройки` рядом с основными действиями сценария.

По нажатию открывается компактный popover с вертикальной формой. В нём находятся:

- короткое описание;
- привязка к должности;
- привязка к сотруднику/кандидату;
- режим запуска;
- конкретная карточка.

Практический смысл:

- правая панель остаётся для редактирования выбранного шага;
- центральная колонка остаётся для структуры сценария, а не для метаданных;
- настройки всего сценария доступны с root-уровня через явную кнопку;
- при проваливании во вложенные ветки центральная колонка показывает уже контекст ветки/цепочки.

Надпись `Основной поток` в root-view больше не используется. Для списка root-шагов используется нейтральный заголовок `Шаги сценария`.

Правила для popover настроек сценария:

- форма вертикальная, каждое поле отдельной строкой;
- подписи полей идут над контролами и используют тот же стиль, что в панели `Детали`: `text-sm font-semibold text-[var(--color-foreground)]/75`;
- кнопка `Сохранить` находится в верхней строке popover справа;
- `Описание` допускается как textarea, чтобы его можно было растянуть вниз;
- вложенные select-popover внутри popover допустимы, если они используют тот же `SingleSelectPicker` и не вводят новый визуальный паттерн.

Этот кейс важен как антипример: если настройки начинают ломать центральный список шагов, их лучше вынести в отдельный контрол, а не пытаться уплотнять сетку до нечитаемого состояния.

## Текст и типографика

Проект сейчас не строится на “брендовой” типографике.

Нужно:

- использовать спокойную, нейтральную sans-систему;
- держать ясную иерархию заголовков;
- чаще выигрывать за счёт spacing и contrast, а не за счёт большого числа шрифтовых трюков.

Не нужно:

- добавлять декоративные шрифты;
- строить стиль только на uppercase и перегруженном letter-spacing;
- делать очень светлый серый текст ради “красоты” в рабочих админских сценариях.

## Что считается хорошим паттерном для проекта

Хорошо:

- мягкие округлые элементы;
- спокойная зелёная accent-система;
- понятные focus-states;
- широкие рабочие панели;
- минимум визуального шума;
- hover, который помогает, а не развлекает;
- повторное использование локальных `ui/*` примитивов.

Плохо:

- случайные новые цвета для каждого экрана;
- тяжёлые тёмные панели на светлом приложении;
- смешение нескольких систем кнопок;
- резкие hover-анимации;
- тесные формы с мелкими controls;
- возврат к “табличному хаосу” без иерархии.

## Как принимать UI-решения дальше

Если экран делается на новом frontend-слое:

1. Сначала использовать существующие токены и `ui/*` компоненты.
2. Потом повторить shell и spacing-паттерны текущих экранов.
3. Только после этого добавлять новый визуальный паттерн, если существующего реально не хватает.

Если есть сомнение между “быстрее накидать новый стиль” и “вписать в существующий язык”, по умолчанию выбираем второе.
